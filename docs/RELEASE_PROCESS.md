# آلية التعديل والنشر الآمنة

## المرجع الإنتاجي

- فرع `main` يمثل النسخة المقبولة فقط.
- الوسم `v1.0.0-stable` يشير إلى النسخة المرجعية الأولى عند الالتزام `0bd4dab`.
- يمنع نسخ مجلد Backup فوق المشروع أو دمج فرع استعادة كامل دون مراجعة ملف بملف.

## أي تعديل جديد

1. تحديث `main` والتأكد أن مساحة العمل نظيفة.
2. إنشاء فرع باسم يصف التعديل.
3. تعديل الملفات المطلوبة فقط.
4. تشغيل الفحوص المحلية ومراجعة `git diff`.
5. رفع الفرع وفتح Pull Request إلى `main`.
6. انتظار نجاح فحص `Validate project` ومراجعة تبويب `Files changed`.
7. دمج الطلب ثم نشر Firebase Hosting أو Functions حسب نطاق التعديل فقط.
8. عند استقرار إصدار مهم، إنشاء Tag جديد وتحديث `CHANGELOG.md`.

## التعامل مع النسخ القديمة

- تفتح النسخة القديمة في فرع أو Worktree منفصل.
- تقارن مع `main` ولا تُنسخ فوقه.
- ننقل فقط الالتزامات أو الأسطر المطلوبة بعد مراجعتها.
- إذا ظهر تعارض، نحافظ على كود `main` ما لم يثبت أن السطر القديم هو المطلوب.

## التراجع الآمن

- نحدد الالتزام الذي سبب المشكلة.
- ننشئ فرع إصلاح.
- نستخدم `git revert` لذلك الالتزام ونفتح Pull Request.
- لا نستخدم إعادة ضبط قسرية لـ`main` ولا نستبدل المشروع بملف ZIP قديم.

## النسخ الاحتياطية

- GitHub مع Tags هو المرجع الأساسي للكود.
- Git Bundle شهري على Google Drive هو النسخة الإضافية القابلة للاستعادة.
- Firestore يُصدّر بصيغته الأصلية إلى Cloud Storage، ثم يمكن الاحتفاظ بنسخة إضافية خارجية.
